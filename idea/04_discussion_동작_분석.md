## 상세 동작 (단계별)

### 1단계: 입력 데이터 준비 (145~158줄)

`_summarize_company()` 함수가 기업 데이터에서 핵심 지표 추출:

```
사용하는 데이터:
- company_info → 기업명, 내부등급
- financial_statements → 총자산, 총부채, 영업이익, 금융비용

계산하는 것:
- 부채비율 = total_liabilities / total_assets × 100
- ICR = operating_income / finance_cost_interest
```

→ "기업명: 창원정밀 | 내부등급: A | 총자산: 258억 | 부채비율: 45.6% | ICR: 2.66배" 형태의 텍스트 생성

### 2단계: RAG 검색 (402~409줄)

**동시에 2개 검색 수행** (Azure AI Search):

| 검색 | 쿼리 | 목적 |
|------|------|------|
| rag_cases | 기업 요약 텍스트 + 심사역 필터 | 해당 심사역의 **과거 심사 사례** |
| rag_rules | "여신심사 규정 부채비율 이자보상배율 담보 신용등급" | **내규/규정** 문서 |

→ `_build_context()`로 합쳐서 `context` 문자열 생성:
```
[과거 심사 사례]
과거 사례 1 내용 (500자)
---
과거 사례 2 내용 (500자)

[규정]
규정 1 내용 (300자)
---
규정 2 내용 (300자)
```

### 3단계: 독립 분석 — 3명 동시 (277~313줄)

#### 수석 CPA (`_analyze_cpa`)
- **보는 데이터**: `financial_statements` (총자산, 부채, 영업이익, 이자비용) + `company_info`
- **받는 컨텍스트**: RAG 검색된 과거 사례 + 규정
- **분석 항목**: 부채비율, ICR, 영업이익률, 매출 추이, DSCR
- **출력**: 상환재원 + 금리결정 적정성 의견 (500자)

#### 산업 애널리스트 (`_analyze_industry`)
- **보는 데이터**: `company_info` (업종코드) + `loan_request` (대출 목적)
- **받는 컨텍스트**: RAG 검색된 과거 사례 + 규정
- **분석 항목**: 업종 전망, 기술력, ESG, 자금용도 타당성
- **출력**: 업체현황(산업) + 기술금융&ESG + 자금용도 의견 (500자)

#### 평판 리스크 탐지관 (`_analyze_reputation`)
- **보는 데이터**: `company_info`만
- **받는 컨텍스트**: RAG 검색된 과거 사례 + 규정
- **분석 항목**: 소송, 체납, 부정 뉴스, 지배구조
- **출력**: 특이사항 + 미이행 승인조건 의견 (500자)

### 4단계: 교차 평가 — 2명 순차 (316~337줄)

#### 본부 수석 심사역 (`_evaluate_underwriter`)
- **보는 데이터**: 3단계 결과 (CPA + 산업 + 평판 분석) + 규정 컨텍스트
- **하는 일**: 3명의 분석을 **규정·여신 정책 관점**에서 검토
- **출력**: 종합의견 + 미이행 승인조건 보완 (400자)

#### 기업영업 RM (`_evaluate_rm`)
- **보는 데이터**: `loan_request` + CPA·산업 분석 결과
- **하는 일**: 현장 관점 정보 보완 (서류에 안 드러나는 정성 정보)
- **출력**: 현장 보완 의견 (300자)

---

### 5단계: 검증관 반박 (340~348줄)

#### 여신심사 검증관 (`_verify_critic`)
- **보는 데이터**: 5명의 전체 분석 결과 (합쳐서 2000자 제한)
- **하는 일**: 근거 부족, 논리 허점, 누락 항목 지적
- **출력**: 지적 사항 텍스트 + `is_resolved` 플래그
  - "합의 충분" 또는 "이의 없음" 포함 → `is_resolved = True` → 루프 종료
  - 아니면 → 재조사 루프 진입

---

### 6단계: 재조사 루프 (416~482줄)

```python
for round_no in range(1, 4):  # 최대 3회
    # 3~5단계 반복
    if is_resolved:
        break  # 검증관 이의 없으면 종료
    else:
        # 검증관 지적 내용으로 추가 RAG 검색
        extra = await _rag_async(critic_opinion[:200], top=2)
        context += extra  # 컨텍스트 보강 후 다음 라운드
```

**핵심**: 검증관이 "근거 부족"을 지적하면, 그 지적 내용으로 **추가 RAG 검색**해서 context를 보강한 뒤 다시 분석.

6. 기업영업 RM 평가 (300자)

**출력 JSON 형식:**

### 8단계: 프론트 변환 (234~270줄)

`_items_to_frontend()` — 내부 스키마 → 프론트 형식:
- grade: "상"/"중"/"하" (자금용도, 영업점장 특별의견은 등급 없음)
- doc_status: "충족"/"부족"
- can_reinforce_persuasion: 등급이 "중" or "하"이면 true
- can_reinforce_documents: doc_status가 "부족"이면 true

## 데이터 흐름 다이어그램

```
company_data (JSON)
├── company_info ──────────→ CPA, 산업, 평판 (3명 모두)
├── financial_statements ──→ CPA (재무 분석)
├── loan_request ──────────→ 산업 (자금용도), RM (현장)
├── sales_data ────────────→ (현재 미사용 — CPA가 간접 참조)
└── collateral_auction ────→ (현재 미사용 — 규정에서 간접 참조)

Azure AI Search (RAG)
├── 과거 심사 사례 ────────→ context → 모든 agent
└── 여신 규정 ─────────────→ context → 모든 agent

GPT 호출 패턴:
각 agent마다 1회 호출 = system prompt(페르소나) + user prompt(데이터+컨텍스트)
루프 1회 기준: 최소 7번 GPT 호출 (CPA, 산업, 평판, 심사역, RM, 검증관, 의장)
```

## 현재 한계점 (개선 기회)

1. **산업 애널리스트가 실제 산업 데이터를 못 봄** — RAG에 산업 자료 없으면 추측할 수밖에 없음
   → **Bing Grounding / Web Search로 해결 가능**

2. **평판 탐지관이 실제 공시/뉴스를 못 봄** — company_info만 보고 판단
   → **DART API로 해결 가능**

3. **sales_data, collateral_auction이 직접 활용 안 됨** — 의장 합성 시 간접 참조만
   → CPA 프롬프트에 명시적으로 넣으면 개선 가능
